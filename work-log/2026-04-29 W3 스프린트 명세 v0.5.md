## 2. 결함 진단 (Day 2 ship 후 발견)

### 2.1 결함 A — 정량 KPI 통계적 power 부족

- 현재 baseline: 6 doc + 사용자 추가 의지 ~10~20 → 최대 ~26 doc
- 추가 50+ doc 인제스트할 의지·자료 **없음** (사용자 명시 확인)
- 통계적 현실: 후보 ≤26 doc + top-10 ⇒ Recall@10 거의 1.0 자동 도달 (degenerate metric)
- `+5pp` 차이를 통계적으로 잡으려면 N(쿼리)≥80 + N(docs)≥100 (실무 RAG eval 표준)
- 결론: v0.4 KPI 는 **6주 MVP 일정 안에서 측정 불가능**

### 2.2 결함 B — `simple` FTS 한국어 어절 한계

- `plainto_tsquery('simple', '공사대금 합의해지')` → '공사대금' & '합의해지' AND 토큰
- 한국어 조사·어미 변형 (`합의해지` vs `합의해지를` vs `합의해지의`) 은 다른 토큰
- Day 2 smoke 측정: `dense_hits=40, sparse_hits=0` (자연어 쿼리에서 sparse 거의 항상 0)
- 결론: sparse path 가 사실상 작동 안 함 → "+5pp" 가 구조적으로 측정 불가능

### 2.3 결함 C — chunk 품질이 진짜 bottleneck

- v0.4 는 검색 알고리즘 (HNSW + FTS + RRF) 만 다룸
- 실제로는 chunk 입력 품질 (PDF 표 노이즈, 청킹 경계, section_title 정확도) 이 검색 출력 품질의 직접 결정 요인
- 사용자 명시 요구: "검색 정확도는 신경써야 함, chunk 품질 포함 깊이 있는 고민 필요"

검색 출력 품질의 진짜 bottleneck 이 chunk 입력 품질이라는 미반영 지점. 사용자 명시 요구. 자동 필터링 risk 는 R-W3-4 정성 검토 게이트로 통제.
