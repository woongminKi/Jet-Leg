# 2026-04-23 W1 Day 7+ 인제스트 실패 처리 정책

> Day 7 까지의 4 커밋 기반 위에 같은 날짜 (2026-04-23) 로 추가 5 커밋(+ 본 work-log) 실시. 인제스트 파이프라인이 도중에 실패했을 때의 데이터 일관성 + 사용자 회복 경험을 정리.

---

## 1. 결정 배경

Day 5/6 운영 중 관찰된 빈틈
- `embed` 스테이지 NOT NULL 위반(2f11073) 같은 후반부 스테이지 실패 시 chunks 는 INSERT 됐는데 dense_vec 만 비어 검색이 깨지는 부분 일관성 발생
- `documents` row 는 살아있는데 chunks 만 일부/전체 비어있는 상태가 사용자에게 노출 → 검색 결과 0건 / 통계 왜곡
- 실패한 doc 이 들어있는 상태에서 사용자가 같은 파일을 다시 드래그하면 Tier 1 dedup 분기로 `duplicated=true` 응답 → 영원히 회복 불가

이 빈틈을 메우기 위해 사용자와 합의한 **B 안 (Hybrid) + 빈틈 6개 보강** 정책을 5 커밋으로 구현.

## 2. 정책 요약

### 2.1 cleanup (커밋 1)
- 파이프라인 fail 시 `_cleanup_failed_doc(doc_id, error_msg)` 자동 호출
- chunks DELETE + `doc_embedding=NULL`
- `documents.flags` 는 select-then-merge 로 기존 키 보존 + `failed=true` / `failed_at` / `error_msg(<=500자)` 마킹
- documents row 자체와 tag_summarize 결과(`tags`/`summary`) 는 디버깅용으로 보존
- cleanup 자체가 실패해도 fail_job 보고는 보존되도록 try/except 분리

### 2.2 자동 reingest (커밋 2)
- `POST /documents` 의 sha256 dedup 분기에서 매칭된 doc 의 `flags.failed` 가 true 면 `duplicated=true` 응답 대신 `_reset_doc_for_reingest` + 새 ingest_jobs 큐잉
- 응답: `duplicated=false` (실제로 새 처리)
- 사용자는 **별도 액션 없이 같은 파일을 다시 드래그하기만 하면** 자동 재시도

### 2.3 사용자 시야에서 failed 격리 (커밋 3)
- `GET /documents` — 기본 응답에서 `flags.failed` 제외, `?include_failed=true` 옵션으로 디버그 노출
- `GET /stats` — 실패 문서를 모든 집계(by_doc_type/by_source_channel/total_size_bytes/extract_skipped/added_*/popular_tags)에서 제외
- `DocumentsStats.failed_count` 필드 신설 — 실패 회수만 별도 노출

### 2.4 UI 재시도 (커밋 4·5)
- `reingestDocument(docId)` API 함수 + `apiPost` (body 없는 POST) 헬퍼 추가
- `UploadItem` 의 job.status === 'failed' 카드 우측에 RefreshCw 아이콘 + "재시도" 버튼 노출
- 버튼 클릭 → reingest 호출 → 부모(`ingest/page.tsx`) 가 `jobId` 갱신 + `retryNonce++`
- `useJobStatusPolling` 가 deps 에 `retryNonce` 를 포함해 같은 doc_id 라도 새 폴링을 시작

## 5. 사이드 이펙트 점검

| # | 시나리오 | 위험도 | 결론 |
|---|---|---|---|
| 1 | failed 자동 reingest 와 사용자의 명시적 POST /reingest 가 동시 호출 | 낮음 | 둘 다 같은 doc_id 의 새 ingest_jobs row 를 만들지만, `_reset_doc_for_reingest` 가 idempotent 한 reset (chunks DELETE + 메타 NULL). 두 번째 새 job 이 첫 번째 job 의 결과를 덮어쓰는 형태로 수렴 |
| 2 | 자동 reingest 분기에서 `_reset_doc_for_reingest` 가 실패하면 | 낮음 | 호출자가 try/except 미적용 → 500 응답. 사용자가 다시 시도 가능. cleanup 자체가 idempotent 라 데이터 손상은 없음 |
| 3 | `flags.failed` 마킹된 문서가 통계에서 빠져 카운트 불일치 | 의도된 동작 | 사용자 시야에서는 "내 지식 베이스" = 정상 인제스트만. `failed_count` 로 별도 노출 |
| 4 | `include_failed=false` 페이지네이션의 total | 정합 | Python 측에서 후필터링 후 `len(filtered)` 로 total 계산. 리스트의 페이지 합과 일치 |
| 5 | retryNonce 갱신 시 이전 폴링의 stale 응답이 새 상태에 들어옴 | 낮음 | 이전 effect 의 cleanup 함수에서 `cancelled = true` → 늦게 도착한 응답은 setState 호출 자체가 스킵됨 |
| 6 | retryNonce 변경 후 첫 tick 도착 전 이전 timedOut 표시 | 일시적 OK | 첫 tick 응답이 도착하는 즉시 `timedOut: false` 로 reset. 1.5초 (POLL_INTERVAL_MS) 미만의 짧은 깜빡임 |
| 7 | 자동 reingest 가 사용자가 같은 파일을 1회 더 드롭한 것을 트리거하는데, 정작 실패 원인이 외부 의존성(Gemini 한도 등)이라 같은 에러 반복 | 사용자 명시적 액션 한도 내 | 매번 cleanup → 새 job → 다시 fail 사이클이 돌지만 데이터는 항상 일관 상태. RPD 한도 소비는 사용자 가시성에 맡김 |

## 6. 사용자 직접 확인 체크리스트

### 6.1 cleanup + 자동 reingest (백엔드만 검증)
```bash
# 1. 의도적으로 실패할 파일 업로드 (예: 손상된 PDF 또는 빈 PDF)
curl -F "file=@broken.pdf" http://localhost:8000/documents
# → 202 + {"doc_id": "<X>", "job_id": "<J1>", "duplicated": false}

# 2. 약 30초 후 status 확인
curl http://localhost:8000/documents/<X>/status
# → job.status = "failed"

# 3. flags 확인 (include_failed 로 노출)
curl 'http://localhost:8000/documents?include_failed=true' | python3 -m json.tool
# → 해당 doc 의 flags.failed = true / error_msg 존재
# → chunks_count = 0

# 4. 일반 리스트에는 안 나타남
curl 'http://localhost:8000/documents' | python3 -m json.tool
# → 해당 doc 미표시

# 5. 같은 파일 재업로드 → 자동 reingest
curl -F "file=@broken.pdf" http://localhost:8000/documents
# → 202 + {"doc_id": "<X>", "job_id": "<J2>", "duplicated": false}
# → 같은 doc_id, 새 job_id, duplicated=false
```

### 6.2 UI 재시도 버튼
| # | 액션 | 기대 |
|---|---|---|
| 1 | broken.pdf 드래그 → 인제스트 화면 | 카드 노출, 진행률 진행 후 "실패" 배지 + error_msg 박스 |
| 2 | 카드 우측 "재시도" 버튼 노출 확인 | RefreshCw 아이콘 + 텍스트 |
| 3 | 재시도 클릭 | 버튼이 spinner 로 잠깐 회전, 카드가 다시 "처리 중" 으로 전환 |
| 4 | 다시 실패하면 | 새 error_msg 가 카드에 갱신됨, 재시도 버튼 다시 활성 |
| 5 | 정상 PDF 로 다시 드래그(같은 파일명) | 자동 reingest 동작, "중복" 배지 안 뜨고 정상 처리 |

### 6.3 stats 검증
```bash
curl http://localhost:8000/stats | python3 -m json.tool
# → documents.total 에서 failed 제외됨
# → documents.failed_count 신규 필드 노출
```
