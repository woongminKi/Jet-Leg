## 0. 프로젝트 한 줄 소개

> **"정리하지 않아도, 기억의 단편으로 꺼내 쓰는 개인 지식 에이전트."**

PDF / HWP / HWPX / DOCX / PPTX 같은 한국 직장인의 멀티포맷 자료를 적재하고, 자연어 질문으로 검색·답변하는 RAG (Retrieval-Augmented Generation) 시스템. 약 3주 동안 개인 프로젝트로 0 → 11개 적재 → 자동 측정 baseline 확보 → 검색 품질 회복까지 완성.

## 1. 핵심 기능 시연 (꼭지 ①)

### 1.1 한 페이지로 보는 시스템 흐름

```
[업로드]   →    [인제스트 9 stage]    →    [DB 저장]    ←→    [검색·답변]
PDF 등          parser → chunk →                              자연어 query →
HWP/HWPX        embed → tag/summary                            top-N + AI 답변
DOCX/PPTX
```

### 1.2 사용한 모델 (모두 의도적 선택)

| 단계 | 모델 | 비용 | 선택 이유 |
|---|---|---|---|
| **임베딩 (chunk + 검색)** | **BAAI/bge-m3** (HF Inference API) | **무료** | 한국어 multilingual 강함, 1024 dim, dense + sparse 동시 지원 (오픈소스 SOTA) |
| **태그·요약** | **Gemini 2.5 Flash** (무료 tier) | 무료 (RPD 1500) | 한국어 요약 품질 우수, 무료 tier 충분 |
| **이미지/스캔 PDF Vision** | **Gemini 2.5 Flash** (**유료 paid tier**) | **$0.00075/페이지** ⭐ | **유일한 유료 채택** — 무료 RPD 20 한계로 한 PDF 도 처리 못함 → paid tier 만이 본질 해결 |
| **답변 생성 (`/ask`)** | Gemini 2.5 Flash (무료 tier) | 무료 | RAG 답변, citations 포함 |
| **Reranker (cross-encoder)** | **BAAI/bge-reranker-v2-m3** (HF Inference API) | **무료** | 한국어 multilingual cross-encoder, +4.65pp top-1 |
| **검색 적합도 평가** | BGE-M3 cosine 휴리스틱 | 무료 | RAGAS LLM-judge 한국어 false negative 회피 |

→ **이미지 분석만 유료 (한 곳)**. 나머지 모두 무료 모델로 운영. **무료 한계 인정 → 본질만 유료** 의 원칙.

### 1.3 멀티포맷 파서 (5 doc_type 처리)

| 확장자 | Parser | page 정보 | 핵심 도전 |
|---|---|---|---|
| PDF | PyMuPDF + Gemini Vision (스캔/그림) | ✅ 페이지 | 표 cell 순서 / 그림 OCR — vision_enrich 로 해결 |
| HWP (5.x) | olefile + bodytext stream 직접 파싱 | ❌ | 한컴 OLE 포맷, 표 병합 휴리스틱 |
| HWPX | XML namespace 파싱 | ❌ | XML 표 행 단위 추출 |
| DOCX | python-docx | ❌ | 단락 + 표 + 헤딩 |
| PPTX | python-pptx | ✅ 슬라이드 번호 | 텍스트 0 슬라이드 vision rerouting |

**11 docs 적재 완료**: 차량 카탈로그 / AI 정책 / 브랜딩 / 경제전망 / 의료 정책 / 학습자료 (태양계/삼국시대) / 법규 (직제·운영내규) / 법률 판례 — 5 doc_type, chunks 5~1045 범위.

### 1.4 하이브리드 검색 (Dense + Sparse + RRF)

```
사용자 query "쏘나타 시트 종류"
       ↓
[A] BGE-M3 임베딩 (1024-dim, LRU cache)
       ↓                              ↓
[B] Dense 검색                 [C] Sparse 검색
   pgvector cosine                PGroonga 한국어 형태소 (Mecab)
   chunks.dense_vec               chunks.text "&@~" 매칭
   rank_dense 1~50                rank_sparse 1~50
            ↓
[D] RRF Fusion (Reciprocal Rank Fusion)
   score = 1/(60 + rank_dense) + 1/(60 + rank_sparse)
            ↓
[E] 표지 가드 (짧은 표지 chunk 패널티 ×0.3)
            ↓
[F] doc 단위 그룹 + 페이지네이션 + 응답
```

→ Dense 단독의 한계 (표지/짧은 텍스트 우세) + Sparse 단독의 한계 (동의어 못 잡음) **상호 보완**.

### 1.5 시연 가능한 기능

| 기능 | URL | 예시 |
|---|---|---|
| **자료 업로드** | `/ingest` | drag-and-drop, 5 doc_type |
| **실시간 진행** | header indicator | Realtime + 60s safety resync |
| **자연어 검색** | `/search?q=...` | "쏘나타 시트 종류" / "체육관 휴관일" / "변제충당 순서" |
| **AI 답변 생성** | `/ask?q=...` | RAG 답변 + 신뢰도 배지 + 출처 [1] [2] 클릭 |

| **자동 골든셋 측정** | `make eval-retrieval` | R@10/MRR/nDCG + multi-doc top-K hit |

## 2. 어려웠던 기술 챌린지 (꼭지 ②)

### 2.1 PDF 표/그림 정확도 한계 → 유료 모델 결정

**문제** (2026-05-04): 정부 데이터센터 정책 PDF 41 페이지 적재. 사용자 query "테스트베드 사업 체계 다이어그램" → 답변 부정확. p.6 의 다이어그램이 chunks 0건.

**진단**:
- PyMuPDF 가 표 cell 순서 뒤섞임 + 이미지 블록 무시
- 41 페이지 중 그림·표 페이지 9건 정보 누락

**해결 시도 4회 정정**:
1. `find_tables()` API + page 확장 → 표만 해결, 그림 누락 (오버 스코프 X)
2. heuristic D (3 룰 OR) → 임계값 미검증 over-engineering
3. 모든 페이지 vision (무료) → quota RPD 20 발견 = 1 PDF 도 불가
4. **유료 Gemini Flash paid tier ($0.03/PDF) 채택** → vision 41/41 (100%) 달성

→ **무료 한계를 인정하고 본질만 유료**. 비용 미미 ($3/100 PDFs).

### 2.2 한국어 sparse 검색의 본질적 한계

**문제**: "소나타 시트 종류" query 가 sparse path 0 hits. `Sonata` (영문) 만 본문에 있고 `소나타` 0건.

**해결 흐름** (W25 D9~D14):
- (a) PGroonga `&@~` AND → OR 변환 (한 토큰만 매칭해도 sparse 결과 진입)
- (b) 한국어 조사 strip whitelist ("전폭은" → "전폭")
- (c) **D2 도메인 동의어 사전 expansion** (W25 D14+1) — "쏘나타 ↔ Sonata", "AI ↔ 인공지능", "휴관 ↔ 휴무" 30 entry. **multi-doc top-1 +2.32pp**.

### 2.3 NFD/NFC 정규화 사고

**문제** (W25 D14): macOS Finder 가 한국어 파일명을 NFD (자모 분리) 로 보냄. DB title 은 NFC. byte 매칭 fail → "승인글 템플릿1" 삭제 시 발견.

**해결**:
- 인제스트 단 `title` NFC 통일
- `/search`·`/answer` query NFC 정규화
- D1 sprint 에서 chunks.text + doc_embedding 입력도 NFC (인제스트단 통일)

### 2.4 Reranker endpoint 함정 — 정량 측정으로 발견된 회귀

**문제** (W25 D14+1):
- BGE-reranker-v2-m3 ship 했으나 정량 측정 시 **-50% 회귀** 발견
- HF API `sentence-similarity` pipeline 이 cross-encoder 가 아니라 일반 sentence embedding cosine 으로 동작 (query-independent ranking)

**진단** — curl 직접 호출 검증:

| query | 시트 chunk | 안전 chunk | 날씨 chunk |
|---|---:|---:|---:|
| "시트 종류" (sentence-similarity) | 0.7304 | **0.9758** ❌ | 0.6468 |
| "시트 종류" (text-classification + [SEP]) | **0.9982** ✅ | 0.0182 | 0.0000 |

**해결**: endpoint 를 `text-classification` + `{q} [SEP] {p}` schema 로 변경 → cross-encoder 정상 작동. **multi-doc top-1 +4.65pp / doc-scope MRR +6.01pp** 회복.

→ **정량 baseline 없이 ship 한 위험**. 정성 smoke 만으론 발견 불가.

### 2.5 측정 인프라 narrowness false negative

**문제** (QA 진단): 골든셋이 정답 chunk 를 `expected_chunk_idx_hints` strict set 으로 평가. 사용자 의도엔 적합하지만 narrow 정답 set 에 없으면 fail 처리.

**해결** (D1):
- `acceptable_chunks` 컬럼 도입 (graded relevance, weight 0.5)
- BGE-M3 cosine ≥ 0.7 chunks 자동 acceptable 라벨
- self-supervised goldenset (Gemini chunk → query 역생성) — 사용자 부담 0

## 5. 무료 모델 사용 철학 (사용자 의도)

> **"무료 모델의 한계를 극복하고 최대한의 효율을 내고 싶었다."**

### 5.1 의도적 선택 표

| 분류 | 모델 | 무료/유료 | 결정 근거 |
|---|---|---|---|
| 임베딩 (BGE-M3) | HF Inference API | 무료 | LRU cache + 싱글톤 + retry 로 quota 안정 운영 |
| Reranker (BGE-reranker-v2-m3) | HF Inference API | 무료 | endpoint 변경 (text-classification + [SEP]) 으로 회복 |
| 태그·요약 (Gemini Flash) | Gemini 무료 tier | 무료 | RPD 1500 충분, 한국어 품질 우수 |
| 답변 생성 (Gemini Flash) | Gemini 무료 tier | 무료 | citations 처리 우수 |
| **이미지/PDF Vision** | **Gemini paid tier** | **유료 ⭐** | **무료 RPD 20 한계 → 1 PDF 도 불가능 → paid tier 만 본질 해결** |
| 검색 적합도 평가 | BGE-M3 cosine 휴리스틱 | 무료 | RAGAS LLM-judge 한국어 false negative 회피 |

### 5.2 무료 한계 극복 패턴 (4가지)

1. **LRU cache** — 같은 query/chunk 반복 시 외부 호출 0 (BGE-M3 / Reranker / HyDE 모두 적용)
2. **graceful fallback** — 외부 API 503/transient 시 sparse-only / RRF / 원본 query 로 silent degradation 회피
3. **싱글톤 + retry** — httpx.Client 누수 차단 + 5xx/network transient 분류 + backoff
4. **opt-in ENV** — 비용·latency 가산 후보는 default off, 사용자 명시적 활성

→ 결과: **무료 모델로 11 docs 적재 + 검색 baseline R@10 0.71 / multi-doc top-1 0.91 / 답변 faithfulness 0.96** 달성.

## 6. 시연 추천 (실 데모 흐름)

### 6.1 권고 순서 (10~15분)

1. **(2분) 프로젝트 한 줄 소개 + 5 doc_type 적재 화면**
   - "한국 직장인의 멀티포맷 자료를 자연어로 꺼내쓰는 RAG"
   - `/ingest` 페이지 — 11 docs (PDF/HWP/HWPX/DOCX/PPTX) 카드 보여주기

2. **(3분) 검색 데모 — 표지 가드 + Hybrid RRF**
   - **query 1**: `소나타 시트 종류 뭐가 있어?`
     - top-1 doc: sonata catalog ✅
     - matched chunk: chunk[44] p.22 (Colors 표) — **표지 chunk 회피** 강조
     - 매칭 강도 100% + 출처 [1] [2] [3]
   - **query 2**: `체육관 휴관일 언제예요`
     - top-1 doc: 한마음생활체육관_운영_내규 ✅
     - hwpx 도 정상 검색 강조
   - **query 3**: `변제충당 순서`

## 7. 예상 질문 10개 + 답변 가이드

### Q1. **왜 ChatGPT / Notion AI 같은 도구 안 쓰고 자체 RAG 시스템을 만들었나?**

**A**: 한국 직장인 페르소나 (공공·대기업 비-IT) 의 두 제약이 핵심.
1. **외부 클라우드 차단** (회사 보안 정책) — Notion AI / ChatGPT 외부 업로드 불가
2. **HWP/HWPX 본질 약함** — 기존 도구는 한국어 한컴 포맷에 약함
→ 자체 RAG 가 본질 해결. 추후 자체 호스팅 배포 가능.

### Q2. **무료 모델로 어떻게 한국어 검색 정확도를 확보했나?**

**A**: 4가지 패턴:
1. **BGE-M3** (한국어 multilingual SOTA) + **PGroonga Mecab** Hybrid RRF
2. **LRU cache + 싱글톤** 으로 quota / latency 절약
3. **graceful fallback** (sparse-only / RRF) 으로 외부 API 일시 실패 흡수
4. **정량 측정 + endpoint 검증** (Reranker fix 사례)
→ 결과: multi-doc top-1 hit 0.91, 답변 faithfulness 0.96.

### Q3. **사용한 핵심 모델 알려달라.**

**A**:
- **임베딩**: BGE-M3 (BAAI, 1024 dim, HF Inference API, 무료)
- **Vision**: Gemini 2.5 Flash (paid tier, $0.00075/페이지)
- **LLM (요약/답변)**: Gemini 2.5 Flash (무료 tier)
- **Reranker**: BGE-reranker-v2-m3 (HF Inference API, 무료)
→ **이미지 분석만 유료**, 나머지 모두 무료. 의도적 비용 최소화.

### Q4. **하이브리드 검색이란? 왜 그렇게 만들었나?**

**A**: Dense (의미 유사도) + Sparse (키워드 매칭) 두 path 의 RRF (Reciprocal Rank Fusion):
- Dense 만: "쏘나타 시트" 의미 매칭 정확하지만 짧은 표지 chunk 우세
- Sparse 만: 키워드 매칭 정확하지만 동의어 ("Sonata" vs "쏘나타") 못 잡음
→ **상호 보완**. 한국어 RAG 표준 패턴.

### Q5. **5가지 doc_type 어떻게 처리하나?**

**A**: 9 stage 인제스트 파이프라인의 `extract` 단계에서 확장자별 parser 분기:
- PDF: PyMuPDF + Vision (스캔/그림)
- HWP: olefile + bodytext stream 직접 파싱 (한컴 5.x OLE)
- HWPX: XML namespace 파싱
- DOCX: python-docx
- PPTX: python-pptx
이후 chunks 단위 통일 → DB 저장. 검색은 doc_type 무관 동일 흐름.

### Q6. **표/그림 인식 정확도는?**

**A**: 데이터센터 PDF (41 페이지) 기준 **vision_enrich 41/41 (100%) 달성**. p.6 다이어그램 같은 그림 정보도 OCR + 의미 추출. 단 PDF 만 적용 — HWP/HWPX/DOCX 의 그림은 미적용 (후속 sprint).

### Q7. **검색 성능을 어떻게 측정했나?**

**A**: 자체 측정 인프라 구축:

- 11 docs × 4 chunks = 43 query
- BGE-M3 cosine ≥ 0.7 chunks 자동 acceptable 라벨 (graded relevance)
- `make eval-retrieval` → R@10 / MRR / nDCG@10 + multi-doc top-K hit
→ baseline 확보 후 5 후보 효과 정량 측정.

### Q8. **Reranker 회귀를 어떻게 발견했나?**

**A**: 정성 smoke 만 했을 때는 좋아 보였음. 그러나 **자동 골든셋 위에서 정량 측정 시 -50% 회귀** 발견.

**교훈: 정량 baseline 없이 ship 위험. smoke 정성만으론 발견 불가.**

### Q9. **한국어 형태소 처리는 어떻게?**

1. **PGroonga Mecab 토크나이저** (DB 단)

4. **도메인 동의어 사전** (W25 D14+1 D2) — 30 entry, 외래어/약어 양방향

1. **default on 결정** — reranker (가장 큰 효과 +4.65pp) 운영 활성화 결정 (latency +2~3s trade-off)

**A**: 자동 골든셋 위에서의 **multi-doc doc-level top-1 hit 비율**.

| 항목 | 값 |
|---|---|
| 측정 대상 | 자동 골든셋 **43 query** (11 docs × 4 chunks 각 × Gemini 가 query 1개 자동 생성) |
| 측정 방식 | 사용자가 검색했을 때 (`doc_id` 미지정 일반 검색), **top-1 으로 나오는 doc 가 정답 doc 와 일치하는 비율** |

> "**43개 자동 query 중 39개가 정답 doc 를 검색 결과 1위로 정확히 찾음**". 3 query 중 약 2.7 개가 top-1 hit.

**의미하지 *않는* 것** (정직):
- ❌ 답변 정확도 91% 가 아님 (별도 — RAGAS faithfulness 0.96 / context precision 0.60)

- ❌ 사용자 만족도 아님 (사용자 실 query 측정 안 됨)

- **N=43 → 95% 신뢰구간 ±5~7pp** (paper 급 ±2pp 아님)
- 자동 골든셋의 query naturalness — Gemini 가 chunk 보고 생성한 query 라 사용자 실제 query 패턴과 다를 수 있음

**정확한 표현 권고**:
> "자체 측정 인프라로 자동 생성한 43건 골든셋 기준, 검색 결과 top-1 doc 가 정답 doc 와 일치하는 비율 = **90.7% (≈ 91%)**. 통계 신뢰구간 ±5~7pp 로 약 85~96% 범위는 N 확장 후 다음 검증 단계."

### Q12. **"한국어 검색 성능 향상을 위해 어떠한 방법으로 했나?"**

#### A. 기본 인프라 (W3, 2026-04-29)

  - **AND → OR 변환** (W25 D10) — 한 토큰만 매칭해도 sparse 결과 진입 → RRF 가산 가능
  - **조사 strip whitelist** (W25 D11) — "전폭은" → "전폭" 으로 변환 후 매칭

#### C. 표지 청크 가드 (W25 D4)
- 짧은 표지 chunk (`chunk_idx=0` AND `page=1` AND `text_len ≤ 30`) → score × 0.3 패널티
- SONATA 카탈로그 p.1 의 6자 표지 ("SONATA") top-1 진입 회피

#### D. NFC 정규화 (W25 D14 + D14+1 D1)

#### E. 도메인 동의어 사전 (W25 D14+1 D2)
- `query_expansion.py` 30 entry — 쏘나타↔Sonata, AI↔인공지능, 휴관↔휴무
- multi-doc top-1 **+2.32pp**

#### F. doc-level embedding RRF 가산 (W25 D14+1 D3)
- chunks 매칭만 보던 점수에 doc 요약 임베딩 매칭도 가산 (k=10/weight=2.0)
- multi-doc top-1 **+2.32pp**

#### G. HyDE (W25 D14+1 D4)
- Gemini 가 query → 가상 답변 1문단 생성 → embedding 보강

#### H. Cross-encoder Reranker (W25 D14+1 D5)

- **multi-doc top-1 +4.65pp** (가장 큰 효과)

→ **누적 결과**: multi-doc top-1 0.9070 (default), reranker 활성 시 0.9535.

### Q13. **"리랭커 도입 이후 오히려 성능이 떨어지는 현상이 발생했을 때 어떻게 해결했나?"**

**A**: 6 단계 — **정량 측정 → 패턴 분석 → 가설 → curl 검증 → endpoint 변경 → 회복**.

#### 1단계: 회귀 발견
S2 (BGE-reranker-v2-m3 ship, sentence-similarity pipeline) 직후 **정량 baseline 위 측정**:

| 메트릭 | OFF (RRF) | ON (reranker) | Δ |
|---|---:|---:|---:|
| Recall@10 | 0.9000 | 0.4000 | **-50%** |
| MRR | 0.7167 | 0.1192 | **-60%** |
| nDCG@10 | 0.7458 | 0.1856 | **-56%** |

**ship 시 정성 smoke 만 했을 때는 좋아 보였음**. 정량 baseline 위에서야 회귀 확인.

#### 2단계: 패턴 분석 — query 무관 동일 ranking
ON 시 top-5 가 거의 모든 query 에서 같은 chunks (12, 6, 16, 45, 43):

| query | OFF top-5 | ON top-5 |
|---|---|---|
| 시트 종류 | **47, 46, 44**, 42, 53 | 53, 12, 6, 45, 43 |
| 안전 기능 | **30, 27, 26**, 61, 29 | **12, 53, 6**, 16, 45 |

→ **query-independent ranking** (정상 cross-encoder 라면 query 마다 score 달라야 함).

#### 3단계: 가설
HF Inference API `sentence-similarity` pipeline 이 BGE-reranker-v2-m3 cross-encoder 가 아니라 일반 sentence embedding cosine 으로 동작.

#### 4단계: curl 직접 검증

| query | 시트 chunk | 안전 chunk | 날씨 chunk |
|---|---:|---:|---:|
| "시트 종류" | 0.7304 | **0.9758** ❌ | 0.6468 |
| "안전 기능" | 0.6890 | 0.9614 ✅ | 0.6037 |
| "날씨 어때" | **0.9795** ❌ | 0.8180 | 0.9863 |

→ "시트 종류" query 에 안전 chunk 가 더 높은 score = 명백한 실패. 가설 확정.

#### 5단계: endpoint 변경 — text-classification + `[SEP]` schema

```python
url = "https://router.huggingface.co/hf-inference/models/BAAI/bge-reranker-v2-m3/pipeline/text-classification"
body = {"inputs": ["{query} [SEP] {passage}", ...]}
```

검증:
| query | 시트 chunk | 안전 chunk | 날씨 chunk |
|---|---:|---:|---:|
| "시트 종류" | **0.9982** ✅ | 0.0182 | 0.0000 |
| "날씨 어때" | 0.0001 | 0.0000 | **0.0316** ✅ |

→ **query-dependent ranking 정상 작동** — 진짜 cross-encoder.

#### 6단계: 정량 회복 + robust 파싱

| 측정 | 회귀 시 | 수정 후 | baseline 대비 |
|---|---:|---:|---:|

추가: HF API 가 batch size 별 응답 형식 다름 (작은 batch flat / 큰 batch nested) → robust parser 추가.

#### 핵심 교훈
> **정량 baseline 없이 ship 위험. 정성 smoke 만으로는 -50% 회귀 발견 불가.** 외부 API endpoint 의 pipeline schema 가 모델 종류와 안 맞을 수 있음 → 직접 검증 필수.

### Q15. **"`/ask` 의 AI 답변 — 무슨 AI? 어떻게 처리?"**

**A**: **Google Gemini 2.5 Flash** (무료 tier) + **RAG (Retrieval-Augmented Generation)** 패턴.

#### 1. 적용 모델 (코드: `api/app/routers/answer.py`)

```python
_LLM_MODEL = "gemini-2.5-flash"
```

답변 흐름 안에서 함께 동작하는 모델:
| 단계 | 모델 | 역할 | 비용 |
|---|---|---|---|
| 1. query 임베딩 | BGE-M3 (HF API) | dense vector 1024-dim | 무료 |
| 2. 검색 (RPC) | PGroonga Mecab + pgvector | Hybrid RRF | 무료 (DB 내장) |
| 3. **답변 생성** | **Gemini 2.5 Flash** | RAG 답변 작성 | **무료 tier** |
| 4. (선택) 답변 평가 | RAGAS Faithfulness / Answer Relevancy | 정량 측정 | Gemini judge |

→ 답변 생성 LLM 만 Gemini Flash. 검색은 무료 self-hosted.

#### 2. 처리 흐름 (5단계 RAG)

```
사용자 질문 "쏘나타 시트 종류 뭐가 있어?"
            ↓
[1] NFC 정규화 (한국어 자모 결합형 통일)
            ↓
[2] _gather_chunks()  ← /search 와 동일 RPC
    BGE-M3 임베딩 + PGroonga query → search_hybrid_rrf
    → top-K (default 5, max 10) chunks 수집
            ↓
[3] _build_messages()  ← 한국어 prompt 구성
    system: "검색 결과 외 추측 금지 + 출처 [N] 인라인"
    user: 질문 + 번호 매겨진 chunks 본문 ([1] / [2] / ...)
            ↓
[4] Gemini 2.5 Flash 호출 (temperature=0.2, faithfulness 우선)
            ↓
[5] AnswerResponse 반환
    answer: LLM 답변 텍스트
    sources: top-K chunks 메타 [{chunk_id, doc_id, doc_title, page, score, snippet}]
```

#### 3. 핵심 prompt — faithfulness 보장

```
당신은 사용자의 개인 지식베이스에서 검색된 자료를 바탕으로 한국어로 답변하는 어시스턴트입니다.
다음 규칙을 반드시 지키세요:
1. 답변은 반드시 제공된 '검색 결과' 안의 내용만 사용하세요. 외부 지식이나 추측을 절대 추가하지 마세요.
2. 검색 결과에 답변할 내용이 없으면 '제공된 자료에서 해당 정보를 찾지 못했습니다.' 라고만 답하세요.
3. 답변 문장 끝에 출처 번호를 [1], [2] 와 같이 인라인으로 표시하세요.
4. 한국어로 간결하게 답변하세요 (5문장 이내 권장).
```

→ **이게 RAG 의 핵심**: LLM 이 검색 결과 외 학습 데이터로 답하면 환각. 위 prompt 가 환각 차단.
→ 측정 결과 **RAGAS Faithfulness 0.959** (95.9% 답변이 검색 결과 내용으로만 작성).

#### 4. 안전 가드

| case | 처리 |
|---|---|
| 검색 결과 0건 | **LLM 호출 안 함** → "제공된 자료에서 해당 정보를 찾지 못했습니다." (quota 보호) |
| Gemini quota 소진 | 503 + Retry-After 1시간 |
| HF transient 실패 | sparse-only 검색 fallback (검색 자체는 진행) |

#### 5. 데모 시연 시 권고 표현

> "검색 → 답변 2단계 RAG 입니다. 먼저 BGE-M3 임베딩 + PGroonga 한국어 검색으로 사용자 자료에서 top-5 chunks 를 찾고, 그 chunks 만 Gemini 2.5 Flash 한국어 prompt 에 넣어서 답변을 만듭니다.
> 핵심은 prompt 에 **'검색 결과 외 추측 금지'** 를 명시해서 환각을 차단한 점이고, 측정 결과 답변의 96% 가 실제 검색 결과 내용 안에서 작성됐습니다."
