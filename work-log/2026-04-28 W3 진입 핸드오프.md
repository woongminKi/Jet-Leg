### 5.3 Day 2 의 핵심 설계 결정점

**`/search` 응답 schema** (backward compatible 보장):
```python
class SearchResponse:
    items: list[SearchItem]   # 기존 (변경 X)
    total: int                # 기존 (변경 X)
    query_parsed: QueryParsedInfo  # 신규 (S2 §7 투명성)

class QueryParsedInfo:
    has_dense: bool
    has_sparse: bool
    dense_hits: int
    sparse_hits: int
    fused: int
```

**라우터 흐름**:
1. `q` 입력 → `BgeM3HfEmbedding.embed_query(q)` → `dense vector(1024)` (1회 HF API)
2. RPC `search_hybrid_rrf(q, dense, k=60, top_k=50)` 호출 → top 50 chunks + scores
3. 메타 필터 (tags/doc_type/from_date/to_date) WHERE 적용
4. doc_id 별 그룹화 — max(rrf_score) + 매칭 청크 최대 3건
5. `SearchResponse` 응답

**엣지 케이스** (Day 2 구현 시 챙길 것):
- HF API timeout → P95 위협 → MVP 는 cache 없이, 실패 시 sparse-only fallback
- 빈 dense/sparse 둘 다 0건 → `total: 0, items: []`
- `q` 길이 200자 초과 → 400
- `dense_vec NULL` chunks 가 있어도 RPC crash 없음 (이미 RPC 안에서 `WHERE dense_vec IS NOT NULL` 처리)
