## 4. 한국어 적합성 — 약점 11건 (P0~P3)

### P0 (검색 품질 직접 영향, Day 3~4 처리 권고)

#### 4.1 [P0] 종결어미 정규식의 lookbehind fixed-width 한계

- **현상**: Python `re` 의 lookbehind 는 fixed-width 만 허용. `(?<=[다요까죠습니다])` 는 character class 안에 5글자 `습니다` 가 들어있는데, **character class 는 단일 char 매칭**이라 실제로는 `다·요·까·죠·습·니·다` 7개 단일 글자 lookbehind 로 동작.
- **위치**: `chunk.py:30-32`
- **영향**:
  - "습니다" 라는 종결어미 자체를 lookbehind 로 잡지 못함 (이미 `다` 가 잡으므로 결과적으로 OK)
  - 그러나 `안습니다` `같습니다` 같은 정상 종결을 정확히 어절 단위로 잡는 게 아니라 **마지막 `다` 만으로 split** → 약어 `대다수 안.` 같은 case 에서 `다` 뒤 `.\s` split 가능
  - 또한 `~한다.` `~된다.` 등 일반 종결은 `다` 매칭으로 정상 작동 — 결과적으로는 OK 한 케이스가 많음
- **권장 (W4-Q-14)**:
  - 대안 1: `re.compile` 의 lookbehind 를 어절 단위 패턴으로 — `r"(?:다|요|까|죠|습니다|입니다|합니다)\.\s+"` 같은 alternation (lookbehind 없이)
  - 대안 2: 형태소 분석기 (`konlpy`/`soynlp`) 도입 — 의존성 추가, 정책 위반
  - **MVP 권장**: lookbehind 제거 + alternation 으로 재작성 (의존성 0, 정확도 ↑)

#### 4.2 [P0] False split — 숫자·약어·"등" 등의 over-segmentation

- **현상**: `(?<=[.!?])\s+` 가 숫자 후 `.` (e.g. `Section 1. Introduction`), 약어 (`vs.`, `et al.`), 한국어 "등." 등에서 의미 단위 한가운데 split
- **위치**: `chunk.py:30` 첫 alternation
- **영향**:
  - `미국·일본·한국 등.` → split → "미국·일본·한국 등" + 다음 문장. 의미 일관 OK 한 case
  - `Section 1. Introduction\n본문...` → "Section 1" + "Introduction\n본문..." → 제목과 본문 분리. **의미 손실**
  - `대법원 2025. 7. 9. 선고 2024다74413 판결` → "대법원 2025" + "7" + "9. 선고 2024다74413 판결" → **법률 인용 파편화**
- **권장 (W4-Q-14)**:
  - lookbehind 에 negative pattern 추가 — `(?<![0-9A-Za-z])\.\s+` (숫자/영문 직후 `. ` 는 분할 안 함)
  - `법령 인용 패턴 (`yyyy. m. d. 선고 ...`) 보호 — 법률 doc 비중 높은 사용자 자산 고려

#### 4.3 [P0] 따옴표·괄호 안 문장 split

- **현상**: 따옴표/괄호 내부의 `.` 를 외부와 동일하게 split
- **위치**: `chunk.py:30` (lookahead/behind 가 따옴표 인식 X)
- **영향**:
  - `대법원은 "공사대금 합의해지를 인정한다." 라고 판결했다.` → "대법원은 ... 인정한다" + "라고 판결했다" → **인용문이 문맥에서 분리**
  - `(주: 단, 예외 있음.)` → 괄호 안 split → 괄호 짝 깨짐
- **권장**:
  - 빠른 fix: 따옴표 짝 검사 — 청크 경계가 odd-count 따옴표면 다음 짝까지 확장 (lookahead 1단계)
  - 본격 fix: 형태소 분석기 의존 (정책 위반) → MVP 는 빠른 fix 만
