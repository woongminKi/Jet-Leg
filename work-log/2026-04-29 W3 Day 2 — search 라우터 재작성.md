## 3. 라우터 흐름 (실제 구현)

1. `q.strip()` (빈 문자열·200자 초과는 400/422)
2. `doc_type` 화이트리스트 검증, `from_date/to_date` ISO 8601 파싱 (실패 시 400)
3. `embed_query(q)` → 1024 dim dense vector
   - 실패 시: `dense_vec = None` 후 `_sparse_only_fallback()` 진입
4. `client.rpc("search_hybrid_rrf", {...}).execute()` → top 50 chunks (각 row: chunk_id · doc_id · rrf_score · dense_rank · sparse_rank)
5. `query_parsed` 집계 — dense/sparse hits, fused = len(rpc_rows)
6. doc_id 별 RRF score 그룹화 (max)
7. 후보 doc_ids 로 `documents` 메타 fetch + 메타 필터 4종 WHERE 적용
8. RRF 내림차순 정렬 → offset/limit 페이지네이션
9. 페이지에 해당하는 doc 들의 매칭 청크 본문 fetch (각 doc 의 RRF top 3, chunk_idx 오름차순으로 표시)
10. `relevance` 정규화 (top=1.0) → `SearchResponse` 응답

## 5. 남은 이슈

### 5.1 (a) Dense path 가 무관 쿼리에도 항상 top_k 반환 — 노이즈 가능성

RPC 의 `dense_hits` CTE 가 `ORDER BY <distance> LIMIT top_k` 라 distance threshold 가 없음. 의미 없는 영문 쿼리 (`zzzznonexistentkeyword`) 도 dense_hits=40, sparse_hits=0 으로 결과 2건 노출.

- 이건 RPC 의 본질이며 일반적 하이브리드 검색 동작.
- 영향: 사용자가 의도와 무관한 쿼리 시 의미 없는 결과 노출 가능.
- 해결 옵션: (1) RPC 에 distance 임계값 (`WHERE c.dense_vec <=> q < 0.5` 같은) 도입, (2) sparse_hits=0 일 때 결과 추가 검증, (3) 그대로 두고 Recall 우선.
- 결정: **Day 5 ablation 측정 후 KPI 영향 보고 검토** (명세 §3.A AC 의 Recall/+5pp 측정에서 결정).

### 5.2 (b) Sparse-only fallback 의 ranking 약화

PostgREST `plfts` 필터는 `ts_rank` 정렬을 노출하지 않아 fetch 순서가 임의. HF API 실패 시 진입하는 fallback 이라 실제로는 드물게 발생하며, 503 보다는 degraded 결과가 낫다는 판단으로 **MVP 수용**. 향후 필요 시 `search_sparse_only` 같은 별도 RPC 를 추가하는 방안.

### 5.3 (c) RPC top_k=50 의 깊은 페이지네이션 한계

RPC 가 항상 상위 50 chunks 만 반환하므로, 그룹화 후 unique doc 50개 이상은 가져올 수 없음. `offset >= 50` 인 페이지는 비게 됨. MVP 단일 사용자·수십 doc 규모에서는 무관 — W5+ 멀티유저 시 RPC 에 OFFSET 파라미터 추가 검토.

### 5.4 (d) `MatchedChunk.highlight` 가 dense-only 매칭에서는 빈 배열

`_make_snippet_with_highlights` 는 q 가 chunk text 에 **리터럴로 포함**될 때만 highlight 를 채움. 하이브리드 RRF 의 dense path 는 의미 매칭이라 리터럴 미포함도 흔함 → highlight=[] 인 케이스가 많아짐. 프론트 ResultCard 가 빈 highlight 도 안전하게 렌더하는지 W3 Day 3+ UI 검수 시 확인.
