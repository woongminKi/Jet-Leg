## 1. 오늘 달성한 것

Day 5 는 목적별로 Phase A / B 로 쪼개 진행.

### Phase A — 외부 API 없이, 검증 UX 강화

Day 4 종료 시점 남은 이슈 2건 해소 + 사용자 직접 검증 수단 추가.

- **커밋 A1** `fix(ingest)`: `tag_summarize.duration_ms` 측정 누락 해소. `begin/end` 직접 호출 경로에 `time.monotonic()` 쌍 추가
- **커밋 A2** `fix(ingest)`: `ingest_logs` `started_at` 을 Python `_now_iso()` 로 통일. DB `DEFAULT now()` 와의 시계 skew 로 인한 `finished_at < started_at` 역전 해소
- **커밋 A3** `feat(api)`: `GET /documents` 리스트 엔드포인트 — 각 문서의 tags·summary·flags·chunks_count·latest_job_status 한 번에
- **커밋 A4** `feat(api)`: `GET /stats` + `GET /` → `/docs` 리다이렉트. 브라우저로 루트 열면 자동 Swagger UI, `/stats` 는 시스템 스냅샷(문서·청크·jobs·실패 샘플)

### Phase B — HF 토큰 받은 후, 임베딩 파이프라인

- **커밋 B1** `feat(api)`: `Settings` · `.env.example` 에 `HF_API_TOKEN` 추가
- **커밋 B2** `feat(adapters)`: **`BGEM3HFEmbeddingProvider`** — `EmbeddingProvider` Protocol 구현체
  - 2026 년 유효 endpoint: `router.huggingface.co/hf-inference/models/BAAI/bge-m3/pipeline/feature-extraction`
  - 구 `api-inference.huggingface.co/models/...` 는 현재 404 (Inference Providers 라우터로 이주)
  - dense 1024-dim 만, sparse 는 `{}`. BGE-M3 lexical weights 는 HF Inference API 가 노출하지 않아 확보 불가. **W3 에 Postgres FTS 로 sparse 대체 검토**
  - 배치 호출 + 3회 retry + 지수 백오프 (cold start 대비 base delay 5s)
- **커밋 B3** `feat(ingest)`: **`embed` 스테이지**
  - `dense_vec IS NULL` 청크만 조회 → 16개 단위 배치 임베딩 → Supabase `upsert(on_conflict=id)`
  - 재처리 엔드포인트 (`POST /documents/{id}/reingest`) 는 W2 예정
- **커밋 B4** `feat(ingest)`: **`doc_embed` 스테이지** — `documents.doc_embedding` 생성
  - 소스 우선순위: summary+implications → raw_text[:3000] → 스킵
  - Tier 2/3 dedup 전제
- **커밋 B6** `feat(ingest)`: **`dedup` 스테이지** — Tier 2/3 중복 감지
  - 기획서 §10.8 그대로: sim ≥ 0.95 (Tier 2) / sim ≥ 0.85 + filename ≥ 0.6 (Tier 3)
  - Python 측 cosine 계산 (MVP 수백 문서 규모)
  - 파일명 유사도: `difflib.SequenceMatcher`
  - 검출만 수행, UI 경고·머지·diff 호출은 W3
- **커밋 B5** `feat(api)`: **`GET /search`** 간이 키워드 검색
  - Postgres `ilike` 기반 리터럴 부분 문자열 일치
  - documents!inner 조인으로 user_id·deleted_at 필터
  - 스니펫: 매칭 위치 ±80자 + 생략기호(…)
  - 하이브리드 dense·sparse·RRF 는 W3

### 파이프라인 최종 형태 (Day 5 말)

```
POST /documents
  ↓ 입력 게이트 A + SHA-256 dedup (Tier 1) + Storage 업로드 + documents insert
BackgroundTasks.add_task(run_pipeline, job_id, doc_id)
  ↓
extract → chunk → tag_summarize → load → embed → doc_embed → dedup (Tier 2/3) → finish
```

### 2.2 Phase B 개별 검증

- **BGE-M3 어댑터**: single `Jet-Rag 한국어 RAG 테스트` → 1024-dim 벡터. batch `["문장 A", "문장 B", "문장 C"]` → 3 × 1024
- **파이프라인 스테이지 코드**: import smoke 전부 OK. 실 데이터 E2E 는 아래 "사용자 직접 확인" 섹션에 위임 (새 업로드 또는 기존 문서 재수집 필요)
- **`GET /search?q=반도체`** → total=77 · 스니펫 정상. `q=AI` → total=50 · `q=zzzz` → 0

### 2.3 아직 실 데이터에서 확인 안 된 것

Supabase 에 이미 올라간 문서 4건은 Day 4 완료 시점 데이터라 **`dense_vec` · `doc_embedding` 이 전부 NULL**. 파이프라인 Day 5 스테이지(`embed`·`doc_embed`·`dedup`) 는 **새로 업로드되는 문서** 부터 실 동작. 기존 문서는 재처리 엔드포인트(W2 예정) 또는 삭제 후 재업로드 필요.

Gemini RPD 소비 고려해서 실 E2E 는 사용자 판단에 맡김. 사용자 확인 체크리스트는 §8 에.

### 실 임베딩 파이프라인 테스트 (선택, Gemini RPD 2 + HF 호출 소비)

**옵션 A — 새 한국어 PDF 올리기**
- `assets/` 에 다른 PDF 준비하거나, sample-report.pdf 내용 일부 변경한 사본
- `POST /documents` Execute, 파일 선택
- `GET /documents/{id}/status` 로 파이프라인 전이 관찰: extract → chunk → tag_summarize → load → embed → doc_embed → dedup → done (1~2분)
- 완료 후 `GET /documents` 에서 해당 문서의 `tags`·`summary`·`flags` 채워졌는지

**옵션 B — sample-report 재처리 (기존 문서 삭제 후 재업로드)**
- Supabase SQL Editor:
  ```sql
  DELETE FROM documents WHERE id='3970feab-e02e-4b93-958b-09177a5debb6';
  -- CASCADE 로 chunks·ingest_jobs·ingest_logs 자동 삭제
  ```
- Storage → documents bucket → 해시 파일 (`3970feab…` 아님, 실제 storage_path 는 `<sha256>.pdf`) 삭제
- `/docs` → `POST /documents` → 같은 파일 업로드 → 이번엔 Day 5 파이프라인 전체 돌아감
- Tier 2/3 dedup 검증: 다른 유사 PDF 도 올려보면 `flags.duplicate_tier` 채워지는지 확인

### Phase B 스테이지 수동 검증 (최소 소비)

HF/Gemini 호출 안 하고 Phase B 스테이지만 확인하려면 Supabase SQL Editor 에서:

```sql
SELECT stage, status, duration_ms FROM ingest_logs
WHERE job_id IN (SELECT id FROM ingest_jobs ORDER BY queued_at DESC LIMIT 1);
```

→ 새 업로드 이후 `embed`·`doc_embed`·`dedup` 스테이지 로그 3건이 추가로 보이면 파이프라인 연결 정상.

## 9. 회고 한 줄

Day 5 는 **로직 동작 검증** 이 사용자 직접 수단으로 처음 가능해진 날. `GET /stats` · `/documents` · `/search` 세 엔드포인트로 Swagger UI 에서 시스템 상태·내용·검색을 모두 눌러 확인 가능. 그 위에 BGE-M3 임베딩 · 문서 대표 벡터 · Tier 2/3 감지까지 얹어 **W1 DoD "PDF 업로드 → 키워드 검색 가능"** 를 만족. 실 데이터 전면 E2E 는 사용자 브라우저에 위임.
