## 4. 협업 작업 방식 (메타 패턴) — 다음 세션에 그대로 재사용

### 4.1 3-agent 교차검증 (중요 결정 시)

명세 v0.3 의 항목 M (`/doc/[id]` 경량판) 결정 시 senior-planner + senior-qa + senior-developer 3개 에이전트가 독립 판정 → 전원 1위 일치한 안 채택. 즉흥 판단 반복 실패 (D→E→F′ 연쇄 입장 변경) 후 도입한 패턴.

**적용 시점**: 명세 변경, 우선순위 결정, 기술 선택 (예: pyhwp vs hwplib).

### 4.2 inert vs non-inert 분리 (hot-reload 환경)

```
변경 분류 (위험 비대칭):
- inert    : helper · 어댑터 method 추가 · 새 모듈 (import 0건). hot-reload 시 무영향.
- non-inert: router · stage · DB INSERT — 마이그레이션 미적용 시 즉시 깨짐.

순서: inert 먼저 → 마이그레이션 적용 확인 → non-inert
```

**적용 사례**: Day 2 의 `put_at` 어댑터 + `_input_gate.py` 먼저 → migration 적용 후 → POST 재작성

### 4.3 dry-run 우선 + smoke cleanup 즉시

- 자동화 스크립트는 default dry-run, `--execute` 명시 시에만 실제 호출 (`backfill_extract_skipped.py`)
- 모든 smoke 후 즉시 cleanup (chunks → ingest_logs → ingest_jobs → documents → Storage blob)
- baseline (`5/453/5/17`) 복구로 다음 작업 영향 0

### 4.4 의존성 그래프 순서 commit

각 일별 6~9 커밋을 다음 순서로:
```
1. chore(repo)         — 환경 (포트, gitignore)
2. chore(deps)         — pyproject + uv.lock
3. feat(db)            — 마이그레이션
4. refactor(adapters)  — 공통 헬퍼 분리·어댑터 method 추가
5. feat(routers)       — input gate · url gate (helper)
6. feat(adapters/impl) — 신규 파서 (HwpxParser, ImageParser, ...)
7. feat(ingest)        — pipeline / dispatcher 변경
8. feat(api)           — 라우터 (POST 재작성, /url, /batch-status, /{id})
9. feat(web)           — 프론트
10. docs(work-log)     — 일별 정리
```

### 4.5 단계별 검증 (단위 → 통합 → live)

```
1. 단위 (unit)         — 함수 직접 호출 (예: _detect_pii 단일 텍스트, _validate_magic 9 케이스)
2. import 검증          — 새 모듈이 mainline 에 import 되는지 (서버 reload 전)
3. 통합 (integration)   — 한 흐름 전체 (POST → BG → DB) 합성 데이터로
4. live                 — 실제 외부 호출 (Gemini Vision, Wikipedia URL, 실 HWP)
5. DB 검증              — MCP `execute_sql` 로 직접 row 확인
```

### 4.6 비판적 판단 + 추천 명시

사용자가 "추천이 뭐야?" 또는 "비판적으로 검토해" 요청 시:
- 옵션 N 개 비교표 → 위험·이익 명시 → **단일 추천 + 한 줄 근거**
- 적용: filetype 채택, Pillow 추가, 폴러 batch 통합, HWP 처리 옵션 (hwp5html / olefile / LibreOffice / Java 비교)
