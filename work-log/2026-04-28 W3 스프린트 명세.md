## 2. 결정 보류 4건 사전 조사 + 단일 추천 (DE-56 ~ DE-59)

### 2.1 DE-56 · pgvector 인덱스 — HNSW vs IVFFlat

| 옵션 | 빌드 시간 | 메모리 | 쿼리 latency | recall@10 | 동적 데이터 적응 | 추가 의존성 |
|---|---|---|---|---|---|---|
| (a) **IVFFlat** (현재 001 마이그레이션) | 빠름 | 적음 (O(nd)) | 중 | ~93% (probes 튜닝 시) | **빌드 시점에 고정** — 데이터 추가 시 재빌드 필요 | 없음 |
| (b) **HNSW** | 느림 (~5x) | 많음 (2~5x) | **빠름** | **95%+ default** | **incremental** — 추가 row 자동 graft | 없음 (pgvector 0.5+) |

**근거**: 1000 doc 규모에서 HNSW 메모리 오버헤드 (m=16, ef_construction=64 기준 ~50MB 추정) 는 무시 가능. IVFFlat 의 critical property "build 시점 데이터 분포 고정" 은 페르소나 A 의 점진 업로드 패턴 (W2 baseline 6 doc → 운영 중 100~1000 까지 증가) 과 상극. 한국 RAG 실무 합의: 1k~10M 범위 RAG 는 HNSW 기본값 (Supabase 공식 가이드 동일).

**▶ 단일 추천**: **(b) HNSW 채택**. 추가 의존성 0 + 점진 업로드 패턴 + 1k 규모 메모리 무시 가능 + 검색 P95 SLO 직결.

**구현 디테일** (W3 Day 1 마이그레이션 003):
```sql
DROP INDEX idx_chunks_dense;
CREATE INDEX idx_chunks_dense ON chunks USING hnsw
    (dense_vec vector_cosine_ops) WITH (m = 16, ef_construction = 64);

DROP INDEX idx_documents_embed;
CREATE INDEX idx_documents_embed ON documents USING hnsw
    (doc_embedding vector_cosine_ops) WITH (m = 16, ef_construction = 64);
```
